# curl-mind

A set of LLM tools that give an AI assistant everything it needs to interact with HTTP APIs: discover what endpoints exist, understand the contract, and execute requests via curl.

Ecosystem: `spec-mind` optimizes the spec. `curl-mind` executes it.

---

## Why

LLMs interacting with HTTP APIs have three needs:

1. **Discovery** — what endpoints exist and across which APIs
2. **Contract** — what to send and what to expect back
3. **Execution** — making the actual HTTP call

Feeding raw OpenAPI JSON covers needs 1 and 2 but at a high token cost (2,000-5,000 tokens per API), and provides no execution capability. `curl-mind` solves all three using `spec-mind`'s `.mind` format (80%+ token reduction) and curl as the execution primitive.

No existing tool does this for LLMs. Static curl generators (Swagger UI, Scalar, cURL Request Generator) produce one-shot commands for humans. `curl-mind` gives the LLM a live, dynamic, multi-environment-aware API execution layer.

### Why not an MCP server?

An MCP server for an API requires building and running a dedicated server process per API, exposing hardcoded tools like `create_payment` or `list_products`. `curl-mind` takes the opposite approach:

- **One tool set for all APIs.** Drop a `.mind` file for any API and the same 3 tools cover it immediately. No new code, no new deployment.
- **Zero infrastructure.** No process to run, no port to expose, no server to keep alive.
- **Token efficiency.** An MCP server floods the LLM context with one tool definition per endpoint. `curl-mind` exposes 3 tools always and fetches only the contract the LLM actually needs.
- **Transparent execution.** curl is inspectable — you always know exactly what was sent. An MCP server hides the HTTP layer behind a function call.
- **Spec-grounded.** The LLM reads the actual contract before making a call, reducing hallucination of endpoints or parameters.

## How it works

`curl-mind` reads `.mind` files from a configured specs folder. Each `.mind` file is a token-optimized representation of an OpenAPI spec, generated by `spec-mind`. The LLM uses three tools in sequence:

```
list_endpoints → get_endpoint_schema → execute_request
```

This mirrors how a developer works: browse what's available, look up the specific endpoint, make the call.

---

## Specs folder

No configuration file. Drop your OpenAPI specs in a folder and let `curl-mind` manage the rest.

```
specs/
  ecommerce.yaml
  payments.yaml
```

Generate `.mind` files using `spec-mind` with the `--no-notation` flag:

```bash
spec-mind sync --no-notation ./specs/
```

**Why `--no-notation`:** The NOTATION legend is identical across every `.mind` file. With multiple APIs loaded, repeating it wastes tokens on every `get_endpoint_schema` call. Instead, `curl-mind` includes the notation once — injected as a fixed prefix in the `get_endpoint_schema` tool description — so the LLM has it in context without redundancy.

The notation that gets injected once:

```
NOTATION: ? optional  [ro] readOnly  [w] writeOnly  =val default
          ^ header  ~cookie  *N multipleOf N  | enum or nullable
          OneOf<A,B> on field = discriminated union
          {*:T} = map/dict  {...} = open object  extends = allOf
          & = inline extension  ~~name~~ = deprecated  # = inline note
          [multipart] [form] [binary] [text] = request body encoding
```

Commit the generated `.mind` files alongside your source specs.

**Base URL resolution:**

- Single server spec: read from `# API: Title — https://base.url/v1` in the `.mind` header
- Multi-server spec: read from `# Servers: Dev=https://..., TST=https://..., Prod=https://...` — matched by `env` label (case-insensitive)
- Auth headers: the caller's responsibility, passed via the `headers` argument on `execute_request`

## Session initialization

When the tools load, an asynchronous indexer is triggered immediately. Tool calls that arrive before the index is ready await its completion — no call is dropped, and no tool needs to know whether the index is fresh or cached.

The indexer:

1. Scans all `*.mind` files in the specs folder concurrently
2. For each file, parses:
   - `api` name (filename without extension)
   - Base URL from `# API: ... — <url>`
   - Server environment map from `# Servers: Label=url, ...` (if present)
   - All `## Section` → `METHOD /path [auth: ...]` entries
3. Resolves into an in-memory index:

```ts
interface EndpointIndex {
  environments: string[]           // deduplicated server labels across all specs
  apis: Record<string, ApiMeta>    // keyed by api name
  endpoints: EndpointEntry[]
}

interface ApiMeta {
  filePath: string                 // absolute path to .mind file
  defaultUrl: string               // from # API header
  servers: Record<string, string>  // label → url from # Servers
}

interface EndpointEntry {
  api: string
  section: string
  method: string
  path: string
  auth?: string
}
```

The index is a `Promise` that resolves once — subsequent accesses return the cached result immediately. `list_endpoints` queries it with no file I/O. `get_endpoint_schema` uses `apis[api].filePath` to open the right file. `execute_request` uses `apis[api].servers[env] ?? apis[api].defaultUrl` to resolve the base URL.

## Tools

### `list_endpoints`

Lists all available endpoints across all loaded `.mind` files. Also surfaces the available environment names derived from each spec's server list, so the LLM knows what values to pass to `execute_request`.

**Input:**

| Field | Type | Required | Description |
|---|---|---|---|
| `filter` | string | No | Substring match on method, path, or section name |

**Output:**

```json
{
  "environments": ["dev", "tst", "prod"],
  "endpoints": [
    { "api": "ecommerce", "section": "Products", "method": "GET",  "path": "/products" },
    { "api": "ecommerce", "section": "Orders",   "method": "POST", "path": "/orders" },
    { "api": "payments",  "section": "Payments", "method": "POST", "path": "/payments", "auth": "oauth2 payments:write" }
  ]
}
```

**How the handler works:**

Queries the pre-built index — no file I/O. Filters `endpoints` by substring match against method, path, and section if `filter` is provided. Returns `environments` deduplicated across all loaded specs.

Call this first when the user references an API or asks what's available.

---

### `get_endpoint_schema`

Returns the full request/response contract for a specific endpoint in `.mind` notation, including all referenced named schemas. Use this before `execute_request` to understand exactly what to send.

**Input:**

| Field | Type | Required | Description |
|---|---|---|---|
| `api` | string | Yes | API name matching the `.mind` filename, e.g. `payments` |
| `method` | string | Yes | HTTP method, e.g. `POST` |
| `path` | string | Yes | Endpoint path, e.g. `/payments` |

**Output:**

The `.mind` block for the endpoint followed by all schemas it references:

```
POST /payments  [auth: oauth2 payments:write]
  ^X-Idempotency-Key:uuid, ^?X-Request-Id:uuid
  body: CreatePaymentRequest
  -> 201: Payment
     headers: Location:uri[ro]
  -> 400: ValidationErrorResponse
  -> 422: ErrorResponse

CreatePaymentRequest:    {amount:float(>=0.01), currency:USD|EUR|GBP|JPY, method:PaymentMethod, metadata?:{*:string}}
Payment:                 {id:uuid[ro], amount:float, currency:USD|EUR|GBP|JPY, status:pending|processing|completed|failed|refunded[ro], createdAt:datetime[ro]}
ValidationErrorResponse: {error, message, fields:{*:string[]}}
ErrorResponse:           {error, message, code?}
```

**How the handler works:**

Uses `index.apis[api].filePath` to open the target `.mind` file. Finds the `METHOD /path` block and extracts it in full (params, body, responses). Collects all uppercase schema names referenced in the block and fetches their definitions from `## Schemas`, recursing one level for schemas that reference other schemas.

Call this after `list_endpoints` and before `execute_request`.

---

### `execute_request`

Executes an HTTP request via curl and returns the status code, response headers, and body. Provide only the relative path — the base URL is resolved from the `.mind` header.

**Input:**

| Field | Type | Required | Description |
|---|---|---|---|
| `api` | string | Yes | API name matching the `.mind` filename, e.g. `payments` |
| `method` | string | Yes | `GET`, `POST`, `PUT`, `PATCH`, or `DELETE` |
| `path` | string | Yes | Relative path, e.g. `/payments/abc-123` |
| `env` | string | No | Server environment label, e.g. `tst`. Uses the first listed server if omitted. |
| `headers` | object | No | Additional request headers, e.g. `{ "Authorization": "Bearer ..." }` |
| `body` | string | No | JSON string body for POST/PUT/PATCH |
| `query_params` | object | No | Key-value pairs appended as query string |

**Output:**

```json
{
  "status": 201,
  "headers": { "Location": "/payments/xyz", "Content-Type": "application/json" },
  "body": "{\"id\":\"xyz\",\"status\":\"pending\"}"
}
```

**How the handler works:**

1. Resolve base URL from the index: `apis[api].servers[env] ?? apis[api].defaultUrl`
2. Build full URL: `baseUrl + path + (query_params ? "?" + encode(query_params) : "")`
3. Build curl args as an array (never interpolated into a shell string)
4. Run via `Bun.$`, capturing response with `-i` flag
5. Parse response into status, headers, body

**Security:**

- The LLM provides only `path`, `headers`, `body`, and `query_params` — never the host
- `path` is validated: must start with `/`, no `..` segments, no protocol prefixes
- curl args are passed as an array, never concatenated into a shell string

Call this after `get_endpoint_schema` with a fully-formed request.

## Agent loop example

```
User: "Create a payment of $50 in the tst environment"

→ list_endpoints(filter: "payment")
  ← { environments: ["dev","tst","prod"], endpoints: [{ api: "payments", method: "POST", path: "/payments", auth: "oauth2 payments:write" }] }

→ get_endpoint_schema(api: "payments", method: "POST", path: "/payments")
  ← .mind block with CreatePaymentRequest schema, required headers, response shapes

→ execute_request(api: "payments", method: "POST", path: "/payments", env: "tst", headers: { "Authorization": "Bearer ..." }, body: '{"amount":50,"currency":"USD","method":{...}}')
  ← { status: 201, body: '{"id":"abc","status":"pending"}' }

LLM: "Payment created. ID: abc, status: pending."
```

## LLM context setup

`curl-mind` ships a `curl-mind.md` file. Add one line to your project's `CLAUDE.md` or `AGENTS.md`:

```
@curl-mind.md
```

This injects the full curl-mind context — tools, workflow, conventions, and the NOTATION legend — once per session. Works identically for Claude Code and OpenCode, no platform-specific wiring needed.

`curl-mind.md` covers:
- The 3 tools and their purpose
- The call sequence: `list_endpoints → get_endpoint_schema → execute_request`
- The NOTATION legend (the single place it lives — stripped from all `.mind` files via `--no-notation`)
- Conventions: never construct URLs manually, always check available environments first, pass auth via `headers`

## Platform registration

The three tool definitions (name, description, input schema, handler) are identical across platforms. Only registration differs.

### OpenCode

Three tool files in `.opencode/tools/`, each using `tool()` from `@opencode-ai/plugin`. Shared parsing and curl logic lives in `_lib/` (not exposed as a tool).

```
.opencode/tools/
  _lib/
    loader.ts      ← builds EndpointIndex from .mind files at load time
    executor.ts    ← curl invocation via Bun.$
  list_endpoints.ts
  get_endpoint_schema.ts
  execute_request.ts
```

### Claude API

Pass the 3 definitions in `tools[]` on every `messages.create` call. Handle `stop_reason: "tool_use"` by routing the `tool_use` block's `name` to the matching handler and returning a `tool_result` block.

```ts
tools: [
  { name: "list_endpoints",      description: "...", input_schema: { ... } },
  { name: "get_endpoint_schema", description: "...", input_schema: { ... } },
  { name: "execute_request",     description: "...", input_schema: { ... } },
]
```

## .mind parsing reference

The handlers parse `.mind` files using these structural rules.

**Line types:**

```
# Source: ...                          skip
# API: Title — https://base.url/v1    default base URL (after " — ")
# Servers: Dev=https://..., TST=...   multi-env server map
[auth: bearer]                         global auth (informational)

## SectionName                         section header
GET /path/{id:uuid}  [auth: none]      endpoint line
  ?page:int=1, ^?X-Id:uuid            params (2-space indent)
  body: SchemaName                     request body
  -> 200: Shape                        response
     headers: Location:uri[ro]         response headers (5-space indent)

## Schemas                             schemas section marker
SchemaName:   {fields...}              schema definition
```

**Index builder (runs at load time):**
- Section: `## Name` line, excluding `## Schemas`
- Endpoint: `/^(~~)?[A-Z]+(~~)? \/\S+/` — strip `~~` wrappers for deprecated methods, strip `{param:type}` annotations from paths, extract `[auth: ...]` if present
- Servers: `# Servers: Label=url, Label=url` — split on `, `, then `=`

**`get_endpoint_schema` parser (runs per call):**
- Endpoint block: from matching `METHOD /path` line to next blank line preceding a new `METHOD` or `##`
- Schema collection: uppercase tokens in the block, excluding `OneOf`, `AnyOf`, `extends`; matched against `## Schemas`; one level of recursion for referenced schemas
