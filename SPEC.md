# api-mind

A set of LLM tools that give an AI assistant everything it needs to interact with HTTP APIs: discover what endpoints exist and understand the contract.

Ecosystem: `spec-mind` optimizes the spec. `api-mind` provides the context. The LLM constructs and executes requests.

---

## Why

LLMs interacting with HTTP APIs have three needs:

1. **Discovery** ŌĆö what endpoints exist and across which APIs
2. **Contract** ŌĆö what to send and what to expect back
3. **Execution** ŌĆö making the actual HTTP call

Feeding raw OpenAPI JSON covers needs 1 and 2 but at a high token cost (2,000-5,000 tokens per API). `api-mind` solves all three using `spec-mind`'s `.mind` format (80%+ token reduction). It exposes 3 tools: `list_apis`, `list_endpoints`, `get_endpoint_schema`.

The LLM constructs and executes curl commands using the `bash` tool, ensuring full transparency.

No existing tool does this for LLMs. Static curl generators (Swagger UI, Scalar, cURL Request Generator) produce one-shot commands for humans. `api-mind` gives the LLM a live, dynamic, multi-environment-aware API discovery layer.

### Why not an MCP server?

An MCP server for an API requires building and running a dedicated server process per API, exposing hardcoded tools like `create_payment` or `list_products`. `api-mind` takes the opposite approach:

- **One tool set for all APIs.** Drop a `.mind` file for any API and the same 3 tools cover it immediately. No new code, no new deployment.
- **Zero infrastructure.** No process to run, no port to expose, no server to keep alive.
- **Token efficiency.** An MCP server floods the LLM context with one tool definition per endpoint. `api-mind` exposes 3 tools always and fetches only the contract the LLM actually needs.
- **Transparent execution.** curl commands executed via `bash` are fully inspectable ŌĆö you always know exactly what was sent.
- **Spec-grounded.** The LLM reads the actual contract before making a call, reducing hallucination of endpoints or parameters.

## How it works

`api-mind` reads `.mind` files from a configured specs folder. Each `.mind` file is a token-optimized representation of an OpenAPI spec, generated by `spec-mind`. The LLM uses three tools in sequence:

```
list_apis ŌåÆ list_endpoints ŌåÆ get_endpoint_schema ŌåÆ [LLM constructs curl] ŌåÆ bash
```

`list_apis` is optional when the API is already known. `list_endpoints` is the typical first step when the API name is known but endpoints are not. `get_endpoint_schema` provides the endpoint contract including base URL, auth, and schemas. The LLM then constructs a curl command and executes it via the `bash` tool.

This mirrors how a developer works: browse what's available, look up the specific endpoint, make the call ŌĆö with full visibility into what's being sent.

## Specs folder

No configuration file. Drop your OpenAPI specs in a folder and let `api-mind` manage the rest.

```
specs/
  ecommerce.yaml
  payments.yaml
```

Generate `.mind` files using `spec-mind` with the `--no-notation` flag:

```bash
spec-mind sync --no-notation ./specs/
```

**Why `--no-notation`:** The NOTATION legend is identical across every `.mind` file. With multiple APIs loaded, repeating it wastes tokens on every `get_endpoint_schema` call. Instead, `api-mind` includes the notation once ŌĆö injected as a fixed prefix in the `get_endpoint_schema` tool description ŌĆö so the LLM has it in context without redundancy.

The notation that gets injected once:

```
NOTATION: ? optional  [ro] readOnly  [w] writeOnly  =val default
          ^ header  ~cookie  *N multipleOf N  | enum or nullable
          OneOf<A,B> on field = discriminated union
          {*:T} = map/dict  {...} = open object  extends = allOf
          & = inline extension  ~~name~~ = deprecated  # = inline note
          [multipart] [form] [binary] [text] = request body encoding
```

Commit the generated `.mind` files alongside your source specs.

**Base URL resolution:**

- Single server spec: read from `# API: Title ŌĆö https://base.url/v1` in the `.mind` header
- Multi-server spec: read from `# Servers: Dev=https://..., TST=https://..., Prod=https://...`
- The LLM sees all available environments and chooses which to use
- Auth headers: the LLM's responsibility, passed in the curl command

## Session initialization

When the tools load, an asynchronous indexer is triggered immediately. Tool calls that arrive before the index is ready await its completion ŌĆö no call is dropped, and no tool needs to know whether the index is fresh or cached.

The indexer:

1. Scans all `*.mind` files in the specs folder concurrently
2. For each file, parses:
   - `api` name (filename without extension)
   - Base URL from `# API: ... ŌĆö <url>`
   - Server environment map from `# Servers: Label=url, ...` (if present)
   - All `## Section` ŌåÆ `METHOD /path [auth: ...]` entries
3. Resolves into an in-memory index:

```ts
interface EndpointIndex {
  environments: string[]           // deduplicated server labels across all specs
  apis: Record<string, ApiMeta>    // keyed by api name
  endpoints: EndpointEntry[]
}

interface ApiMeta {
  filePath: string                 // absolute path to .mind file
  title: string                    // from # API: <title> ŌĆö <url>
  defaultUrl: string               // from # API header
  servers: Record<string, string>  // label ŌåÆ url from # Servers
}

interface EndpointEntry {
  api: string
  section: string
  method: string
  path: string
  auth?: string
}
```

The index is a `Promise` that resolves once ŌĆö subsequent accesses return the cached result immediately. `list_endpoints` queries it with no file I/O. `get_endpoint_schema` uses `apis[api].filePath` to open the right file.

## Tools

### `list_apis`

Lists all APIs loaded from the specs folder, with their names, titles, base URLs, and available environments. Use this when you need to know what APIs are loaded or what environments a specific API supports.

**Input:** none.

**Output:**

```json
{
  "apis": [
    { "name": "ecommerce", "title": "E-commerce API", "defaultUrl": "https://api.example.com/v1", "environments": {} },
    { "name": "payments",  "title": "Payments API",   "defaultUrl": "https://api.payments.example.com/v2", "environments": { "dev": "https://dev.payments.example.com/v2", "tst": "https://tst.payments.example.com/v2", "prod": "https://api.payments.example.com/v2" } }
  ]
}
```

**How the handler works:**

Queries the pre-built index ŌĆö no file I/O. Returns one entry per loaded `.mind` file using `title`, `defaultUrl`, and `servers` from `ApiMeta`.

Call this first when the user references an API you haven't seen yet.

---

### `list_endpoints`

Lists all available endpoints across all loaded `.mind` files. Also surfaces the available environment names derived from each spec's server list.

**Input:**

| Field | Type | Required | Description |
|---|---|---|---|
| `filter` | string | No | Substring match on method, path, or section name |

**Output:**

```json
{
  "environments": ["dev", "tst", "prod"],
  "endpoints": [
    { "api": "ecommerce", "section": "Products", "method": "GET",  "path": "/products" },
    { "api": "ecommerce", "section": "Orders",   "method": "POST", "path": "/orders" },
    { "api": "payments",  "section": "Payments", "method": "POST", "path": "/payments", "auth": "oauth2 payments:write" }
  ]
}
```

**How the handler works:**

Queries the pre-built index ŌĆö no file I/O. Filters `endpoints` by substring match against method, path, and section if `filter` is provided. Returns `environments` deduplicated across all loaded specs.

Call this first when the user references an API or asks what's available.

---

### `get_endpoint_schema`

Returns the full context for a specific endpoint: the base URL, environments, auth requirements, and request/response schema in `.mind` notation. Use this to understand an endpoint before constructing a curl command.

**Input:**

| Field | Type | Required | Description |
|---|---|---|---|
| `api` | string | Yes | API name matching the `.mind` filename, e.g. `payments` |
| `method` | string | Yes | HTTP method, e.g. `POST` |
| `path` | string | Yes | Endpoint path, e.g. `/payments` |

**Output:**

Text block containing:

```
# API: Payments API
# Base URL: https://api.payments.example.com/v2
# Environments:
#   dev: https://dev.payments.example.com/v2
#   tst: https://tst.payments.example.com/v2
#   prod: https://api.payments.example.com/v2

## Endpoint
POST /payments
Auth: oauth2 payments:write

## Schema
POST /payments
  body: CreatePaymentRequest
  -> 201: Payment

## Schemas
CreatePaymentRequest: {amount:float(>=0.01), currency:USD|EUR|GBP|JPY}
Payment: {id:uuid[ro], amount:float, status:pending|completed}
```

The LLM uses this to:
1. Choose an environment (dev/tst/prod)
2. Construct the full URL: `{base_url}{path}`
3. Add auth headers if needed
4. Build the request body according to schemas
5. Execute via `bash` tool: `curl -X POST https://... -H 'Authorization: Bearer <TOKEN>' -d '...'`

**How the handler works:**

Uses `index.apis[api].filePath` to open the target `.mind` file. Finds the `METHOD /path` block and extracts it in full (params, body, responses). Collects all uppercase schema names referenced in the block and fetches their definitions from `## Schemas`, recursing one level for schemas that reference other schemas.

Call this after `list_endpoints` before constructing a curl command.

```

     body: CreatePaymentRequest
```

## LLM context setup

`api-mind` ships a `api-mind.md` file. Add one line to your project's `CLAUDE.md` or `AGENTS.md`:

```
@api-mind.md
```

This injects the full api-mind context ŌĆö tools, workflow, conventions, and the NOTATION legend ŌĆö once per session. Works identically for Claude Code and OpenCode, no platform-specific wiring needed.

`api-mind.md` covers:
- The 3 tools and their purpose
- The call sequence: `list_apis ŌåÆ list_endpoints ŌåÆ get_endpoint_schema ŌåÆ [curl]`
- The NOTATION legend (the single place it lives ŌĆö stripped from all `.mind` files via `--no-notation`)
- Auth patterns: how to interpret `[auth: ...]` and construct headers
- Conventions: never construct URLs manually, always check environments, use correct auth

## .mind parsing reference

The handlers parse `.mind` files using these structural rules.

**Line types:**

```
# Source: ...                          skip
# API: Title ŌĆö https://base.url/v1    default base URL (after " ŌĆö ")
# Servers: Dev=https://..., TST=...   multi-env server map
[auth: bearer]                         global auth (informational)

## SectionName                         section header
GET /path/{id:uuid}  [auth: none]      endpoint line
  ?page:int=1, ^?X-Id:uuid            params (2-space indent)
  body: SchemaName                     request body
  -> 200: Shape                        response
     headers: Location:uri[ro]         response headers (5-space indent)

## Schemas                             schemas section marker
SchemaName:   {fields...}              schema definition
```

**Index builder (runs at load time):**
- Title + default URL: `# API: <title> ŌĆö <url>` ŌĆö split on ` ŌĆö `, store both
- Section: `## Name` line, excluding `## Schemas`
- Endpoint: `/^(~~)?[A-Z]+(~~)? \/\S+/` ŌĆö strip `~~` wrappers for deprecated methods, strip `{param:type}` annotations from paths, extract `[auth: ...]` if present
- Servers: `# Servers: Label=url, Label=url` ŌĆö split on `, `, then `=`

**`get_endpoint_schema` parser (runs per call):**
- Endpoint block: from matching `METHOD /path` line to next blank line preceding a new `METHOD` or `##`
- Schema collection: uppercase tokens in the block, excluding `OneOf`, `AnyOf`, `extends`; matched against `## Schemas`; one level of recursion for referenced schemas
